Scope and closures

Scope


                    let a = 1; // Global scope: everything registered under
                    // window/script, eg. window.a = 1 (script in case of let)
                    function f1() {
                        let b = 2 + a; // Outer function scope
                        // a is *not* visible here

                        return () => {
                            b += 2;
                            let c = a + b; // Inner function scope
                            return c;
                        };
                        // c is *not* visible here
                    }
                    // b is *not* visible here
                    // c is *not* visible here

Closures

Closures are functions that refer to independent (free) variables (variables that are used locally, but defined in an enclosing scope). In other words, these functions 'remember' the environment in which they were created. 

                    // Global scope: everything registered under window/script,
                    // eg. window.a (script in case of let)
                    let f2 = f1(); // A new closure created!
                    a = f2(); // a and b were modified => window.a = 6, f2.f1.b = 5

                    a = f2(); // a and b were modified => window.a = 13, f2.f1.b = 7

                    let f3 = f1(); // A new closure created! f2.f1 !== f3.f1

Closures


                    let a = 1; // Global scope
                    function f1() {
    let b = 2 + a; // Outer function scope
                        // a is *not* visible here

                        return () => {
        b += 2;
                            let c = a + b; // Inner function scope
                            debugger;
                            return c;
                    
    };
}
                    let f2 = f1();
                    let f3 = f1();

Block-scoped function (ES2015/ES6)


                    {
                        function foo () { return 1 }
                        foo() === 1
                        {
                            function foo () { return 2 }
                            foo() === 2
                        }
                        foo() === 1
                    }

Object.prototype

  • All JavaScript objects inherit the properties and methods from their prototype.
  • Objects created using an object literal, or with new Object(), inherit from a prototype called Object.prototype.
  • The Object.prototype is on the top of the prototype chain.
  • All JavaScript objects (Date, Array, RegExp, Function, ....) inherit from the Object.prototype.

DOM model

HTML DOM

  • All the elements of an HTML document are represented as DOM objects
  • Root object is window.document
  • All DOM objects have properties, methods (and events in case of elements)

Properties vs. Attributes

  • Property is a value stored in a browser's memory (DOM object) and can be changed
  • Attribute is set by developer in HTML and is used by browser as initial value for the corresponding property.
Properties are often referenced as attributes, but the actual attributes are stored as DOM Attribute nodes.

window.document

Represents the web page. 

                    document.getElementById(id)             // Find an element
                    document.getElementsByTagName(name)     // Find elements
                    document.getElementsByClassName(name)
                    document.querySelector(cssSelector)     // Find by css selectors

                    document.createElement(element)         // Creates a new element
                    document.appendChild(element)           // Appends a new child
                    document.write(text)                    // Replace content

Elements and nodes

  • All DOM objects are Node instances.
  • Node.nodeType represents the type of the node.
  • Elements have type Node.ELEMENT_NODE
  • Attributes, comments, text nodes, etc. are also Node instances
  • Attributes are stored in Element.attributes
  • Other nodes are stored in Node.children

Useful properties


                    // element.style allow access to all CSS properties
                    element.style.fontSize = '16px';

From MDN:

A Promise is a proxy for a value not necessarily known when the promise is created. It allows you to associate handlers with an asynchronous action's eventual success value or failure reason. This lets asynchronous methods return values like synchronous methods: instead of immediately returning the final value, the asynchronous method returns a promise to supply the value at some point in the future.

Modules

Why modules?

  • Traditional objects
    • Small units
    • Limited responsibility
    • Defined in global scope → name clashes
  • Modules
    • Composition of multiple objects, constants, etc. under single unit
    • Namespaces to bundle multiple objects → no name clashes
    • Transparent scopes

Modules - Naive approach


                    function Car() { ... }

                    let carModule = {
                        Car : Car,
                        someProperty: 2,
                        ...
                    }
                

                    let myCar = new carModule.Car();
                    let prop = carModule.someProperty;
  • ＋ Namespace to bundle multiple objects
  • － Defined in global scope → name clashes

Modules - IIFE

Immediately-Invoked Function Expression 

                    ;(function(global) {
                        function Car() { ... }

                        global.carModule = {
                            Car : Car,
                            someProperty: 2,
                            ...
                        }
                    })(window); // Or "global" in Node.js or any other object
                

                    let myCar = new carModule.Car();
                    let prop = carModule.somePropery;
  • ＋ Defined in function's scope → no name clashes
  • ＋ Only some members are "exported"
  • － Scripts are loaded and executed on DOM load

  • Synchronous and asynchronous file loading

CommonJS (CJS)

  • Dependencies not known prior to execution/loaded on demand:
    • Dependencies can't be loaded asynchronously (are not known) prior to module body execution
    • require is blocking the task queue
  • Native support in Node.js or MongoDB :)
  • Browser support: Browserify, SystemJS

AMD

  • Asynchronous Module Definition
  • Developed for browsers
  • Does not allow cyclic dependencies

                    // module A (a.js)
                    define(function() {
                        function Car() { ... }
                        return {
                            someProperty: 2,
                            Car: Car;
                        }
                    });
                

                    // module B (b.js)
                    define(['./a.js', 'exports', 'module'],
                        function(moduleA, exports, module) {
                            let myCar = new moduleA.Car();
                            exports.carInstance = myCar;
                            exports.moduleId = module.id;
                        });

AMD

  • Compatible with CommonJS

                    // module B (b.js)
                    define(['./a.js', 'exports'],
                        function(moduleA, exports) {
                            let commonJSModule = require('./commonJSModule.js');
                            // Dynamically (and async.) loaded module
                            require(['./someOtherModule.js'], function (otherModule) {
                                exports.asyncModule = otherModule;
                            });
                        });
                

                    // c.js
                    require(['./someModule.js'], function (someModule) {
                        someModule.doSomething();
                    });

AMD

  • Dependencies are known based on the arrays in define and require:
    • Such dependencies are loaded asynchronously prior to module body execution
      (modules have to be non-cyclic)
    • CommonJS style require is still blocking the task queue
  • Not supported on server side
  • Browser support: RequireJS, SystemJS

UMD

  • Universal Module Definition
  • Supported in all environments

                    (function (root, factory) {
                        if (typeof define === 'function' && define.amd) {
                            // AMD. Register as an anonymous module.
                            define(['exports', 'module', 'someModule'], factory);
                        } else if (typeof exports === 'object'
                                && typeof exports.nodeName !== 'string') {
                            factory(exports, module, require('someModule')); // CommonJS
                        } else {
                            factory((root.moduleID = {}), // Browser globals
                                { exports: root.moduleID, id: ''}, root.someModule);
                        }
                    }(this, function (exports, module, someModule) {
                        ...
                        exports.action = function () {};
                    }));

ES2015/ES6

  • Dependencies are known based on static analysis:

What to use?

  • In development use ES2015!
    • Clear and simple syntax
  • In production use IIFE!

Transpiling for development

  • Babel can produce all module formats from ES2015 (CommonJS or AMD suite best;)
  • Modules are loaded via SystemJS, RequireJS or any other tool:
    • Flexible development
    • Code split into multiple files
    • Faster traspilation
    • Dependency injection
    • etc.

Production

  • Don't use modules in production → slower download times
    • Too many HTTP requests
    • Small files
  • One big bundle is much more faster to load and process
  • Use Webpack 2 or Rollup to create the IIFE result

Tree-shaking

Why ES2015 modules in development?
  • Webpack 2 or Rollup operate over ES2015, AMD or CommonJS modules
  • Based on analysis of ES2015 modules they are able to "tree-shake" the final code → smaller bundles → faster load times
  • Tree-shaking removes not used parts of code
  • With ES2015 we import only parts of modules, AMD and CJS imports everything
